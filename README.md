# aurora-rutoken

Тестовое приложение для ОС Аврора, работающее с токенами **Рутокен ЭЦП 3.0** по **USB** и **NFC**.

Стек: нативный Qt/C++ + QML (Аврора SDK, RPM-пакет). Доступ к токену: PKCS#11 (`librtpkcs11ecp.so`) поверх PC/SC (`pcscd`): USB — через CCID-handler, NFC — через NFC-стек ОС (`nfcd`).

## Сборка

RPM собирается автоматически в GitHub Actions при каждом пуше в `main` (workflow [build-rpm.yml](.github/workflows/build-rpm.yml)). Два независимых чистых matrix jobs разворачивают Аврора Platform SDK **5.2.1.200** и отдельно собирают `armv7hl` и `aarch64`; повторное использование object-файлов другой архитектуры исключено. После сборки CI проверяет RPM arch, ELF machine, dynamic loader и метаданные подписи. Для каждой архитектуры CI также скачивает исходный официальный `ru.rutoken.librtpkcs11ecp-2.19.0.0-1` с `download.rutoken.ru`, сверяет закреплённый SHA-256 и проверяет архитектуру/экспорты библиотеки. Публикация происходит только после успеха обеих архитектур. Локально ничего ставить не нужно.

Собранный RPM подписывается **общедоступной тестовой парой ключей ОМП** (профиль Regular, `ci/keys/`, источник — [документация по подписанию](https://developer.auroraos.ru/doc/sdk/app_development/packaging/package_signing)) утилитой `rpmsign-external` внутри чрута PSDK — без подписи Аврора 5 отклоняет пакет с ошибкой `BadPackageSignature`. Пакеты на тестовом ключе ОС принимает **только при включённом режиме разработчика**.

Готовые RPM: **[релиз ci-latest](https://github.com/code-agent-43824/aurora-rutoken/releases/tag/ci-latest)** — обновляется каждой успешной сборкой. Файлы лежат **отдельными `.rpm` рядом, без архива** и скачиваются прямо с телефона в один тап. CI льёт `.rpm` прямо в релиз из сборочных jobs; промежуточные зазипованные артефакты GitHub Actions намеренно не используются как канал доставки.

## Экран токенов в v0.1

Главный экран — **живой список подключённых токенов**. Класс `TokenWatcher` держит библиотеку `/usr/lib/3rdparty/ru.rutoken.librtpkcs11ecp/librtpkcs11ecp.so` загруженной и раз в ~2 с в рабочем потоке выполняет изолированный цикл `C_Initialize` → `C_GetSlotList(CK_TRUE)` → `C_GetSlotInfo`/`C_GetTokenInfo` → `C_Finalize`. Благодаря свежей инициализации на каждом опросе список **обновляется сам**: воткнули Рутокен по USB — карточка появилась; вынули — исчезла; поднесли к NFC-антенне — появилась. UI перерисовывается только при изменении набора токенов.

Карточка токена: тип подключения (бейдж **USB**/**NFC**), метка, **серийный номер**, модель, версия прошивки, флаги и имя PC/SC-ридера. Тип подключения определяется по имени ридера. Если библиотека не установлена — приложение работает и показывает статус в шапке.

Диагностика PC/SC, NFC и жизненного цикла PKCS#11 (из v0.0.2–v0.0.4) переехала на отдельную страницу — «Диагностика» в меню-шторке.

## Детали токена и вход по PIN в v0.2

Тап по карточке открывает экран деталей: серийный номер, модель, производитель, версии, флаги, имя ридера — и поле **ввода PIN пользователя**. Кнопка «Войти» проверяет PIN через `C_Login(CKU_USER)`; при ошибке показывается состояние по флагам токена (осталось мало попыток / последняя попытка / PIN заблокирован). Проверка выполняется одним изолированным циклом `C_Initialize → C_OpenSession → C_Login → C_Logout → C_CloseSession → C_Finalize` в рабочем потоке. PIN не логируется и обнуляется в памяти сразу после использования.

`C_Initialize`/`C_Finalize` глобальны для процесса, поэтому фоновый опрос списка токенов и операция логина сериализованы **общим мьютексом** — их циклы `init…finalize` не пересекаются.

## Объекты токена в v0.3

После входа по PIN (объекты читаются в той же залогиненной сессии, иначе приватные ключи не видны) доступен экран объектов — **двухуровневая структура**:

- верхний уровень — **сертификаты**; под каждым его **ключи**, сгруппированные по `CKA_ID`;
- у сертификата видно, **есть ли ключ** или он «сам по себе»;
- **ключи без сертификата** показаны на верхнем уровне с `CKA_ID` и `CKA_LABEL`;
- у каждого объекта указан **источник чтения** — пока `PKCS#11` (архитектура заложена под добавление других источников).

Перечисление — `C_FindObjectsInit/C_FindObjects/C_FindObjectsFinal` по классам `CKO_CERTIFICATE`, `CKO_PUBLIC_KEY`, `CKO_PRIVATE_KEY`; атрибуты `CKA_ID`/`CKA_LABEL`/`CKA_KEY_TYPE` через `C_GetAttributeValue`. Тип ключа отображается человекочитаемо (RSA, EC, ГОСТ Р 34.10-2012).

Раскладка ABI-структур PKCS#11 (`CK_INFO`, `CK_SLOT_INFO`, `CK_TOKEN_INFO`, префикс `CK_FUNCTION_LIST`) задана с естественным выравниванием (без Windows-упаковки `pack(1)`) и проверяется `static_assert` отдельно для armv7hl (ILP32) и aarch64 (LP64); значения сверены с реальной официальной библиотекой Рутокен.

Бинарная библиотека не копируется внутрь MIT-пакета приложения. В релизе рядом публикуется **неизменённый официальный RPM Рутокен**: это сохраняет исходный пакет и соответствует условиям распространения неизменённого ПО из разделов 4.3–4.4 [лицензионного соглашения Рутокен](https://www.rutoken.ru/download/license/License_Agreement_Rutoken.pdf). Официальная страница загрузки: [PKCS#11-библиотека Рутокен](https://www.rutoken.ru/support/download/pkcs/).

## Подтверждено на физическом устройстве

На 32-битном телефоне с Авророй 5.2 версия 0.0.2 показала зелёные статусы `nfcd`, NFC-адаптера, `libpcsclite` и `pcscd`. При подключении Рутокен ЭЦП по USB в списке появляется дополнительный PC/SC-ридер `Aktiv Rutoken ECP 00`. Работа непосредственно через PKCS#11 начнётся в v0.0.3.

## Установка на телефон (Аврора 5.x)

1. **Включите режим разработчика** (Настройки → Приложения/Система → Режим разработчика; потребуется учётная запись Аврора ID или локальное подтверждение — зависит от версии ОС). Без него пакет, подписанный тестовым ключом, не установится (`BadPackageSignature`) или не запустится.
2. Разрешите установку недоверенных приложений (Настройки → Безопасность → Недоверенное ПО).
3. Скачайте из [релиза ci-latest](https://github.com/code-agent-43824/aurora-rutoken/releases/tag/ci-latest) **два** RPM своей архитектуры: для 32-битного устройства — оба `*.armv7hl.rpm`, для 64-битного — оба `*.aarch64.rpm` (пакет чужой архитектуры отклоняется с ошибкой `BadPackageArchitecture`).
4. Сначала откройте в «Файлах» и установите `ru.rutoken.librtpkcs11ecp-2.19.0.0-1.<архитектура>.rpm`, затем `ru.codeagent43824.rutokentestapp-0.3.0-1.<архитектура>.rpm`.
5. Запустите приложение. Подключите Рутокен по USB или поднесите к NFC-антенне — карточка токена появится в списке **сама**. Тапните по карточке → детали токена и вход по PIN (PIN пользователя по умолчанию — `12345678`); после входа — кнопка «Объекты токена» с двухуровневым списком сертификатов и ключей. Меню-шторка сверху: «Диагностика» и «Обновить». Альтернатива установке из «Файлов» — `devel-su pkcon install-local <путь-к-rpm>`.
