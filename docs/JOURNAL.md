## 2026-07-22 — Claude (кодинг-агент), сессия 6: этап 4a — последовательность экранов ввода PIN (USB) + перечитывание метки (0.5.0-4)

**Событие:** владелец проверил 0.5.0-3: разблокировка через ex-функцию и смена метки работают. Единственное замечание — **после смены метка не перечитывается** (в деталях/списке остаётся старая). И команда: делать этап 4 — для USB тоже последовательность экранов (введи старый PIN → новый → повтор), для NFC собрать все данные и выполнить всё за одну короткую сессию. Этот этап разбит: **4a (USB-последовательности + перечитывание метки)** сейчас, **4b (PIN-сценарии по NFC)** следующим.

**Перечитывание метки (фикс):** корень — `signatureOf` в `TokenWatcher` учитывала только `slotId|serial|connection`, но не метку. Смена метки не меняла сигнатуру, поэтому `onPolled` выходил рано и `m_tokens`/`tokensChanged` не обновлялись — даже `refresh()` не помогал. Теперь метка входит в сигнатуру. Плюс: `TokenLabelPage` при успехе один раз вызывает `tokenWatcher.refresh()` (немедленный опрос; мьютекс уже свободен — воркер смены метки освобождает `runSessionOp`-локер до `emit finished`). `TokenDetailsPage` показывает **живую** метку: свойство `curLabel` ищет токен по `slotId` в `tokenWatcher.tokens` (привязка перевычисляется по `tokensChanged`), заголовок и дочерние экраны берут `curLabel`. Список USB (`TokensPage`) и так читает `tokenWatcher.tokens` → обновляется сам.

**Последовательность экранов (USB):** `PinChangePage` теперь собирает PIN'ы пошагово, переиспользуя проверенный `PinPadPage` (минимум нового QML — на устройстве QML не тестируется CI, поэтому важно не плодить рискованный код). Механика без гонок: `PinPadPage.accept()` испускает `entered`, затем сам делает `pageStack.pop()`; push следующего экрана прямо в обработчике `entered` привёл бы к тому, что `pop()` закрыл бы уже новый экран. Поэтому продвижение — в `onStatusChanged`, когда страница снова **активна** (предыдущий pad полностью закрылся): при первом показе открываем первый шаг; далее следующий шаг открываем только если значение реально ввели (`pendingAdvance`). Возврат «назад» из pad значение не вводит → не запираем в цикле, просто показываем итог. Итоговый экран (кнопки с точками-масками + «Применить») остаётся для проверки/ручного перевведения.

**Проверки на хосте:** скобки/скобки QML сбалансированы (`PinChangePage` 29/29 `{}`, 103/103 `()`; `TokenLabelPage` 18/18, 26/26; `TokenDetailsPage` 34/34, 51/51); `slotId` в картах — `qulonglong`, обе стороны сравнения `===` в `curLabel` из одного источника (как в существующем `loggedInSlot === modelData.slotId`); `runSessionOp` освобождает мьютекс до `emit finished` (нет дедлока с `refresh`). Новых строк для перевода нет (`Confirm new PIN`, метки PIN уже есть). spec 0.5.0-4.

**Следующий шаг:** зелёный CI → проверка на телефоне (последовательность экранов смены PIN; метка перечитывается) → **этап 4b**: PIN-сценарии по NFC (собрать все данные на экранах без токена, затем всё за одну короткую сессию поднесения; при ошибке — заново).

Указатель резолвится через `QLibrary::resolve("C_EX_UnblockUserPIN")` (typedef `ExUnblockUserPINFn`, session-only; сверено на реальной библиотеке — битая сессия → 0xB3). Сигнатура метода упрощена: `unblockUserPin(slot, soPin)` (без нового PIN). `m_invalidateUserPin=false` — кэш входа пользователя остаётся валидным. UI `PinChangePage` в режиме `unblock`: показывается только поле SO PIN, поля нового PIN/подтверждения и метка несовпадения скрыты; `canApply` не требует совпадения; подсказка «администратор сбрасывает счётчик попыток, сам user PIN не меняется».

**Смена метки — вход пользователя (фикс 101):** `C_EX_SetTokenName` требует аутентификацию **пользователя**, а не SO (я в 0.5.0-2 логинился как SO → `CKR_USER_NOT_LOGGED_IN`/101). Теперь `changeTokenLabel(slot, userPin, label)`: `C_Login(USER)` → `C_EX_SetTokenName` → `C_Logout`. UI `TokenLabelPage`: поле «PIN пользователя» вместо админского, подсказка «смена метки требует PIN пользователя и не стирает токен».

**Проверки на хосте:** `pkcs11_minimal.h` компилируется; скобки QML сбалансированы (`PinChangePage` 23/23 `{}`, 70/70 `()`; `TokenLabelPage` 15/15, 23/23); вызов `unblockUserPin(page.slotId, page.pin1)` (2 арг.) сопоставлен с новой сигнатурой; XML переводов валиден (en + ru: строки `TokenLabelPage`/`PinChangePage`). spec 0.5.0-3.

**CI-прогон (run #28, `7e57deb`) — ЗЕЛЁНЫЙ с первой попытки.** `C_EX_UnblockUserPIN` (резолв + typedef + новая сигнатура `unblockUserPin`) и вход `CKU_USER` в `changeTokenLabel` собрались на armv7hl и aarch64; в релизе `ci-latest` 4 голых `.rpm` 0.5.0-3 (старые 0.5.0-2 удалены).

## 2026-07-21 — Claude (кодинг-агент), сессия 5: v0.5 фиксы — смена user PIN, честный счётчик попыток, смена метки (0.5.0-2)

**Событие:** владелец проверил 0.5.0-1: смена SO PIN и разблокировка работают, **смена PIN пользователя падала с 101** (`CKR_USER_NOT_LOGGED_IN`). Плюс два замечания: показывать честный остаток попыток (а не «мало/последняя»); добавить смену метки токена.

**Фикс смены user PIN:** Рутокен требует вход пользователем перед сменой его PIN. `changeUserPin` теперь: `C_Login(USER, old)` → `C_SetPIN(old, new)` → `C_Logout` (раньше вызывал `C_SetPIN` без входа → 0x101).

**Честный счётчик попыток (vendor):** библиотека экспортирует `C_EX_*` напрямую (проверено `nm -D`). `C_EX_GetTokenInfoExtended(slot, &infoEx)` даёт `ulUserRetryCountLeft`/`ulMaxUserRetryCount`. Размер `CK_TOKEN_INFO_EXTENDED` зависит от арки (LP64 = 184, определено probe'ом: верный размер → CKR_OK, иначе 0x07 ARGUMENTS_BAD; библиотека проверяет размер до наличия токена и не требует входа). Поэтому `exUserPinRetries` **подбирает принятый размер в рантайме** и читает поля по фиксированным смещениям CK_ULONG (12 — max, 13 — осталось) — работает на armv7hl и aarch64 без хардкода размера. `loginErrorMessage` теперь показывает «осталось попыток: N из M» (fallback на качественный индикатор по флагам, если vendor недоступен). Указатели `C_EX_*` резолвятся через `QLibrary::resolve`, прокинуты в воркеры (`runTokenWrite`, `run`).

**Смена метки токена (vendor):** `C_EX_SetTokenName(session, label, len)` (сигнатура/смещение сверены: битая сессия → 0xB3). `changeTokenLabel(slot, adminPin, label)`: `C_Login(SO)` → `C_EX_SetTokenName` → `C_Logout` (требует PIN администратора; токен не стирается). UI `TokenLabelPage` (поле метки + админский PIN через `PinPadPage`), пункт «Сменить метку токена» в меню-шторке `TokenDetailsPage`.

**Проверки на хосте:** vendor-API исследован на реальной библиотеке (`nm -D`, probe размера `CK_TOKEN_INFO_EXTENDED`=184 LP64, `C_EX_SetTokenName` bad-session 0xB3); `pkcs11_minimal.h` компилируется; скобки QML сбалансированы; XML переводов валиден. Qt/QML собирает CI. spec 0.5.0-2.

**Оговорка (устройство):** требование входа SO для `C_EX_SetTokenName` — по документации Рутокена; если на устройстве метка меняется иначе (user-логин/без входа) — поправлю по факту. Счётчик попыток админского PIN аналогично доступен (`ulAdminRetryCountLeft`), пока показываем пользовательский.

**CI-прогон (run #27, `a65d1e3`) — ЗЕЛЁНЫЙ с первой попытки.** Vendor-указатели (`C_EX_GetTokenInfoExtended`/`C_EX_SetTokenName`), пробрасывание в воркеры, `changeTokenLabel` и фикс `changeUserPin` собрались на armv7hl и aarch64; в релизе `ci-latest` 4 голых `.rpm` 0.5.0-2 (старые 0.5.0-1 удалены).

## 2026-07-21 — Claude (кодинг-агент), сессия 5: v0.4.1 доработки NFC — логическое подключение, «Отключить», анимация по состоянию, звуки (0.4.1-3)

**Логическое подключение (`TokenSession`):** добавлены `nfcToken` (дескриптор) + `nfcObjects` (снимок), `nfcConnected`, `suppressedUsb`. `commitNfc(token)` сохраняет дескриптор + снимок текущих объектов; `updateNfcObjects()` обновляет снимок после генерации/импорта по NFC; `disconnectNfc()` забывает NFC-токен. `suppressUsb(serial)` скрывает USB-токен; `retainUsbSlot`→`syncWithTokens(tokens)` теперь и сбрасывает вход при пропаже USB-слота, и снимает подавление с серийников, которых больше нет (переподключил → снова виден). `main.cpp` зовёт `syncWithTokens`.

**UI:** `TokensPage` — USB-список фильтрует подавленные серийники; у каждой USB-карточки кнопка «Отключить» (logout если это залогиненный слот + `suppressUsb`); появилась карточка **логически подключённого NFC-токена** (видна при `nfcConnected`) с кнопкой «Отключить» (`disconnectNfc`), тап → `ObjectsPage(NFC)`. `ObjectsPage` показывает `nfcObjects` для NFC и `objects` для USB (`objectsModel`). Мастер на connect-успехе делает `commitNfc(lastToken)` и открывает объекты; на generate/import — `updateNfcObjects()`.

Мастер биндит `animState` (шаг 4 → removing; обнаружен → connected; иначе searching) и показывает иллюстрацию на шагах 3 и 4.

**Фикс:** pack(1) удалён; static_assert'ы переписаны на естественное выравнивание с ветвлением по `sizeof(CK_ULONG)`/`sizeof(void*)` (armv7hl: flags=36, size=76, C_Initialize=4; aarch64: 40/88/8) — 32-битные значения проверяет компилятор armv7hl прямо в CI. Release пакета поднят до `0.0.3-2`.

**Урок для будущих агентов:** ABI-заголовки нельзя «проверять» static_assert'ами, выведенными из самого заголовка, — сверяться нужно с реальной библиотекой или официальным `pkcs11.h`; pack(1) в PKCS#11 — только Windows.
