# aurora-rutoken

Тестовое приложение для ОС Аврора, работающее с токенами **Рутокен ЭЦП 3.0** по **USB** и **NFC**.

Стек: нативный Qt/C++ + QML (Аврора SDK, RPM-пакет). Доступ к токену: PKCS#11 (`librtpkcs11ecp.so`) поверх PC/SC (`pcscd`): USB — через CCID-handler, NFC — через NFC-стек ОС (`nfcd`).

## Сборка

RPM собирается автоматически в GitHub Actions при каждом пуше в `main` (workflow [build-rpm.yml](.github/workflows/build-rpm.yml)). Два независимых чистых matrix jobs разворачивают Аврора Platform SDK **5.2.1.200** и отдельно собирают `armv7hl` и `aarch64`; повторное использование object-файлов другой архитектуры исключено. После сборки CI проверяет RPM arch, ELF machine, dynamic loader и метаданные подписи. Для каждой архитектуры CI также скачивает исходный официальный `ru.rutoken.librtpkcs11ecp-2.19.0.0-1` с `download.rutoken.ru`, сверяет закреплённый SHA-256 и проверяет архитектуру/экспорты библиотеки. Публикация происходит только после успеха обеих архитектур. Локально ничего ставить не нужно.

Собранный RPM подписывается **общедоступной тестовой парой ключей ОМП** (профиль Regular, `ci/keys/`, источник — [документация по подписанию](https://developer.auroraos.ru/doc/sdk/app_development/packaging/package_signing)) утилитой `rpmsign-external` внутри чрута PSDK — без подписи Аврора 5 отклоняет пакет с ошибкой `BadPackageSignature`. Пакеты на тестовом ключе ОС принимает **только при включённом режиме разработчика**.

Готовые RPM — два канала:
- **[релиз ci-latest](https://github.com/code-agent-43824/aurora-rutoken/releases/tag/ci-latest)** (основной) — **отдельные `.rpm` рядом, без архива**, скачиваются прямо с телефона в один тап;
- **артефакты Actions** каждого запуска (`rutokentestapp-<версия>-<арх>-rpms`) — второй канал; GitHub отдаёт их зазипованной папкой.

## Экран токенов в v0.1

Главный экран — **живой список подключённых токенов**. Класс `TokenWatcher` держит библиотеку `/usr/lib/3rdparty/ru.rutoken.librtpkcs11ecp/librtpkcs11ecp.so` загруженной и раз в ~2 с в рабочем потоке выполняет изолированный цикл `C_Initialize` → `C_GetSlotList(CK_TRUE)` → `C_GetSlotInfo`/`C_GetTokenInfo` → `C_Finalize`. Благодаря свежей инициализации на каждом опросе список **обновляется сам**: воткнули Рутокен по USB — карточка появилась; вынули — исчезла; поднесли к NFC-антенне — появилась. UI перерисовывается только при изменении набора токенов.

Карточка токена: тип подключения (бейдж **USB**/**NFC**), метка, **серийный номер**, модель, версия прошивки, флаги и имя PC/SC-ридера. Тип подключения определяется по имени ридера. Если библиотека не установлена — приложение работает и показывает статус в шапке.

Диагностика PC/SC, NFC и жизненного цикла PKCS#11 (из v0.0.2–v0.0.4) переехала на отдельную страницу — «Диагностика» в меню-шторке.

## Детали токена и вход по PIN в v0.2

Тап по карточке открывает экран деталей: серийный номер, модель, производитель, версии, флаги, имя ридера — и поле **ввода PIN пользователя**. Кнопка «Войти» проверяет PIN через `C_Login(CKU_USER)`; при ошибке показывается состояние по флагам токена (осталось мало попыток / последняя попытка / PIN заблокирован). Проверка выполняется одним изолированным циклом `C_Initialize → C_OpenSession → C_Login → C_Logout → C_CloseSession → C_Finalize` в рабочем потоке. PIN не логируется и обнуляется в памяти сразу после использования.

`C_Initialize`/`C_Finalize` глобальны для процесса, поэтому фоновый опрос списка токенов и операция логина сериализованы **общим мьютексом** — их циклы `init…finalize` не пересекаются.

## Объекты токена в v0.3

Экран объектов — **двухуровневая структура**:

- верхний уровень — **сертификаты**. Описание берётся из **разобранного тела X.509** (`CKA_VALUE`, DER) — **Common Name**, **издатель**, **срок истечения** (`QSslCertificate`; при неудаче разбора — fallback на `CKA_LABEL`);
- сертификаты **показываются без ввода PIN** (это публичные объекты). До входа наличие закрытых ключей неизвестно (строка «ключи показываются после входа по PIN»);
- **после входа по PIN** под каждым сертификатом появляются его **ключи** (сгруппированные по `CKA_ID`), видно «есть ключ»/«сам по себе», а **ключи без сертификата** выводятся на верхнем уровне с `CKA_ID`/`CKA_LABEL`;
- у каждого объекта указан **источник чтения** — пока `PKCS#11` (архитектура заложена под другие источники).

Перечисление — `C_FindObjectsInit/C_FindObjects/C_FindObjectsFinal` по классам `CKO_CERTIFICATE`, `CKO_PUBLIC_KEY`, `CKO_PRIVATE_KEY`; атрибуты `CKA_ID`/`CKA_LABEL`/`CKA_KEY_TYPE`/`CKA_VALUE` через `C_GetAttributeValue`. Приватные ключи видны только в залогиненной сессии, поэтому полный список читается после входа. Тип ключа отображается человекочитаемо (RSA, EC, ГОСТ Р 34.10-2012).

## v0.4 (в работе)

Расширенное ТЗ: генерация ключевой пары, импорт сертификата и **экспорт сертификата**.

Готов **экспорт**: тап по сертификату открывает **экран сертификата** (`CertificatePage`) с разобранными полями (Common Name, издатель, срок, CKA_LABEL, CKA_ID, наличие ключа). Сверху — авроровское меню-шторка с пунктом **«Экспорт сертификата»** → экран экспорта, где пользователь выбирает **формат (PEM или DER)**, **имя файла** (плейсхолдер — Common Name сертификата) и **папку** (плейсхолдер — каталог загрузок). Сохраняется тело сертификата **без закрытого ключа**; расширение добавляется автоматически. Требует разрешения `UserDirs`.

Готова **генерация ключевой пары на токене**: меню-шторка на экране объектов → **«Создать ключевую пару»** (`GenerateKeyPage`). Пользователь выбирает **алгоритм и длину** (ГОСТ Р 34.10-2012 256/512 или RSA 2048/4096), задаёт **метку** и вводит **PIN пользователя**. Генерация идёт в рабочем потоке под общим мьютексом PKCS#11: `C_Initialize → C_OpenSession(R/W) → C_Login → C_GenerateKeyPair → перечитывание объектов → C_Logout → C_CloseSession → C_Finalize`. Параметры ГОСТ (OID `CKA_GOSTR3410_PARAMS`/`CKA_GOSTR3411_PARAMS`) и RSA (`CKA_MODULUS_BITS`/`CKA_PUBLIC_EXPONENT`) — из `docs/RESEARCH.md` §5в; смещение `C_GenerateKeyPair` (№60) и раскладка `CK_MECHANISM` сверены с реальной библиотекой. Обе части пары получают общий случайный `CKA_ID` — по нему импортируемый сертификат «приклеивается» к паре. После успеха список объектов обновляется автоматически.

Готов **импорт сертификата из файла**: меню-шторка на экране объектов → **«Импорт сертификата»** (`ImportCertificatePage`). Файл выбирается через `Sailfish.Pickers` (зависимость изолирована в отдельном `CertFilePickerPage`, поэтому путь можно и ввести вручную), поддерживаются **PEM и DER**. Тело разбирается собственным минимальным DER-парсером (проверен на реальном сертификате сверкой смещений и RSA-modulus с `openssl asn1parse`): извлекаются subject/issuer/serial (для атрибутов `CKA_SUBJECT`/`CKA_ISSUER`/`CKA_SERIAL_NUMBER`) и открытый ключ из `SubjectPublicKeyInfo`. **Приклеивание по открытому ключу:** открытый ключ сертификата сравнивается с открытыми ключами на токене (RSA — по modulus; ГОСТ — по `CKA_VALUE`, в прямом и обратном порядке байт), и при совпадении сертификату присваивается `CKA_ID` найденной пары (иначе `CKA_ID` = SHA-1 от открытого ключа). Импортируется только сертификат (`CKO_CERTIFICATE`, `C_CreateObject`) — без закрытого ключа. Требует входа по PIN (R/W-сессия); смещение `C_CreateObject` (№21) сверено с реальной библиотекой.

## v0.4.1 (в работе) — нормальный ввод PIN и NFC-парадигма

Отдельный **экран ввода PIN** (`PinPadPage`) с **цифровой клавиатурой** (0–9, стереть, показать/скрыть) и переключением в **текстовый режим** — обычную клавиатуру ОС. Переиспользуется входом по USB и (в 0.4.1-2) мастером NFC.

**USB — запоминание PIN:** после первого успешного входа PIN хранится в оперативной памяти (`TokenSession`) и переиспользуется генерацией и импортом **без повторного ввода** — до кнопки **«Разлогиниться»**, отключения USB-токена (сбрасывается по сигналу `TokenWatcher`) или закрытия приложения. Экраны генерации/импорта больше не спрашивают PIN отдельно: если вход не выполнен — предлагают войти, иначе работают по запомненному PIN. PIN не пишется на диск и обнуляется в памяти при сбросе.

**NFC — отдельная парадигма:** главный экран показывает USB-токены живым списком, а NFC — **эфемерным пунктом** «Подключить по NFC» (не подключён, но можно подключить). Тап запускает **мастер** (`NfcConnectPage`): 1) взять токен в руки; 2) ввод PIN (тот же `PinPadPage`); 3) поднести к задней крышке — **встроенная иллюстрация** (контур телефона + зона NFC + пульсирующие волны + подъезжающая карточка-токен) и прогресс, приложение ждёт появления NFC-слота (через `TokenWatcher`) и выполняет операцию; 4) убрать токен, результат. **Запись по NFC** (генерация ключа, импорт сертификата) идёт через тот же мастер — на экранах генерации/импорта при NFC кнопка запускает мастер вместо запомненного PIN. **PIN по NFC не кэшируется** (`TokenSession::nfcRead` — вход и чтение без запоминания); экспорт работает офлайн по снимку и токен не требует.

**Логическое подключение и отключение:** после подключения по NFC токен **остаётся в списке** как логически подключённый (`TokenSession` хранит его дескриптор и **снимок объектов**), поэтому к его сертификатам можно вернуться, выйти и снова войти **без повторного поднесения**. У каждого токена в списке — небольшая кнопка **«Отключить»**: NFC-токен логически забывается (`disconnectNfc`), USB-токен скрывается из списка до физического переподключения (по серийному номеру, `suppressUsb`; подавление снимается, когда токен физически исчезает — `syncWithTokens`). Иллюстрация мастера **реагирует на обнаружение**: при появлении токена анимация останавливается и токен «прижат» к задней крышке (идёт обмен), а после завершения операции показывается, как токен **убирают** от телефона; в моменты соединения/рассоединения проигрываются короткие звуки (вложенные WAV из `sounds/` через QtMultimedia `SoundEffect`, изолировано в `Feedback.qml` под `Loader` — при отсутствии модуля просто тихо). Эфемерный пункт «Подключить по NFC» скрывается, пока NFC-токен логически подключён (работаем с одним одновременно).

## v0.5 (в работе) — управление PIN

Меню-шторка на экране деталей токена → **«Смена PIN пользователя»**, **«Смена PIN администратора»**, **«Разблокировать PIN пользователя»** (`PinChangePage`, режимы `user`/`so`/`unblock`). Данные собираются **прямой цепочкой экранов** без возврата к базовому экрану операции между шагами и без списка введённых значений: тап по пункту меню сразу открывает экран текущего PIN, затем новый PIN, затем повтор нового; **как только введён повтор — операция выполняется** (если новые совпали; иначе новый и повтор запрашиваются заново с подсказкой). Разблокировка — один экран (PIN администратора), затем сброс. Технически: `PinPadPage` получил флаг `autoPop` (по умолчанию `true`); в цепочке смены PIN он `false`, поэтому экран сам не закрывается — навигацию ведёт `PinChangePage` (первый шаг — `push`, следующие — `replace`, над контроллером всегда один экран ввода → возврат за результатом простым `pop()`, без гонок push/pop). У нового PIN — **подтверждение** (защита от опечатки, которая иначе заблокировала бы токен). Операции — изолированный цикл под общим мьютексом (`TokenSession`, helper `runSessionOp`: `C_Initialize → C_OpenSession(R/W) → op → C_CloseSession → C_Finalize`):

- смена PIN пользователя — `C_SetPIN(old, new)`;
- смена PIN администратора — `C_Login(CKU_SO, old)` → `C_SetPIN(old, new)` → `C_Logout`;
- разблокировка PIN пользователя — `C_Login(CKU_SO)` → vendor `C_EX_UnblockUserPIN(session)` → `C_Logout` (**сброс счётчика попыток**, сам PIN пользователя не меняется; в UI вводится только PIN администратора).

Успешная смена/сброс пользовательского PIN сбрасывает запомненный вход (кэш стал невалидным). PIN обнуляются в памяти сразу после использования; коды `CKR_PIN_*` показываются понятным текстом. Смещения `C_InitPIN` (№11) и `C_SetPIN` (№12) сверены с реальной библиотекой. Смена **PIN пользователя** идёт с входом пользователем (`C_Login(USER)` → `C_SetPIN` → `C_Logout`).

**Честный остаток попыток:** при ошибке входа показывается «осталось попыток: N из M» — реальные значения через vendor-функцию Рутокена `C_EX_GetTokenInfoExtended` (`ulUserRetryCountLeft`/`ulMaxUserRetryCount`). Размер `CK_TOKEN_INFO_EXTENDED` различается по архитектурам, поэтому принятый размер **подбирается probe'ом в рантайме** (верный → `CKR_OK`), а поля читаются по фиксированным смещениям — надёжно на armv7hl и aarch64.

**Смена метки токена:** меню-шторка на деталях токена → «Сменить метку токена» (`TokenLabelPage`) — vendor `C_EX_SetTokenName` с входом **пользователя** (`C_Login(USER)`; SO давал `CKR_USER_NOT_LOGGED_IN`/101); токен не стирается. После смены метка **перечитывается**: сигнатура набора токенов в `TokenWatcher` теперь учитывает метку (иначе смена не меняла сигнатуру и список не обновлялся), `TokenLabelPage` вызывает `refresh()` при успехе, а `TokenDetailsPage` показывает живую метку (поиск по `slotId` в `tokenWatcher.tokens`, авто-обновление по `tokensChanged`) — новая метка появляется сразу. Смена метки — пока по USB.

**PIN-операции по NFC — из обзора токена, как у USB.** Логически подключённый NFC-токен открывается тем же экраном деталей (`TokenDetailsPage`), что и USB, с той же **меню-шторкой**: смена PIN пользователя/администратора, разблокировка, смена метки. Отдельной карточки для NFC нет — виртуальный NFC-рутокен один. `TokenDetailsPage` стал connection-aware: для NFC пропускается живое чтение/логин (объекты берутся из снимка `nfcObjects`), а пункты меню передают `connection: "NFC"`; USB-путь не меняется. Сценарии USB и NFC совпадают, у NFC добавляется лишь короткое **поднесение**: собираем все данные на экранах (тем же сборщиком — `PinChangePage`/`TokenLabelPage` с `connection: "NFC"`), затем в одной сессии выполняем операцию (детект NFC-слота и иллюстрация — как в мастере подключения: `NfcHoldAnimation`, `findNfcToken`, `tokensChanged`, `refresh`, звуки `Feedback`); после — «уберите токен» и результат, при ошибке — **«Начать заново»**. После смены метки по NFC снимок обновляется (`setNfcLabel`), поэтому новая метка видна в обзоре и списке.

## v0.6 (в работе) — удаление объектов

**Удаление объекта (USB).** На экране объектов **долгое нажатие** на записи удаляет её. У **сертификата** всегда спрашивается область (диалог `DeleteCertPage`): **только сертификат** или **сертификат вместе с ключами**; у отдельного **ключа** — сразу, с отсрочкой `RemorsePopup` (можно отменить). Тот же пункт **«Удалить сертификат»** есть в pull-down-меню карточки сертификата (`CertificatePage`). `C_DestroyObject` (№23) переведён из заглушки в реальный указатель в таблице функций с проверкой смещения `static_assert`. Удаление идёт `TokenSession::deleteObjectsCached(slot, id, keysToo)` → `runTokenWrite` (вход по запомненному PIN, R/W-сессия, `pkcs11::destroyByCkaId(..., onlyCertificate=!keysToo)`: `C_FindObjects` по `CKA_ID` (+ `CKA_CLASS=CKO_CERTIFICATE`, если только сертификат) → `C_DestroyObject`), затем перечитывание объектов в той же сессии — удалённая запись сразу исчезает из списка. Пустой `CKA_ID` игнорируется (защита от массового удаления).

Если удаление сертификата запущено **до входа** (приватные ключи ещё не видны), диалог предупреждает, что у сертификата **может быть закрытый ключ**, скрытый до ввода PIN-кода. Две кнопки: **«Удалить только сертификат»** — удаляет только публичный объект-сертификат **без входа** (`TokenSession::deleteCertPublic`: R/W-сессия без `C_Login`, `C_DestroyObject` по `CKA_ID`+`CKO_CERTIFICATE`; если токен требует входа — сообщение с предложением ввести PIN-код), и **«Ввести PIN-код»** (вход на отдельном экране). После входа объекты перечитываются, наличие ключа определяется честно и появляется вариант «Сертификат и его ключи».

**Удаление по NFC.** То же долгое нажатие (или пункт «Удалить сертификат» в карточке) на NFC-токене ведёт на `NfcDeletePage` — удаление за **одно поднесение**, по модели прочих NFC-операций: сперва собираем всё без токена (для сертификата — выбор области «только сертификат / сертификат и его ключи»; затем ввод PIN-кода; для ключа область не спрашивается — сразу PIN-код), потом одно поднесение к задней крышке выполняет всё за одну сессию (`TokenSession::deleteObjects(slot, pin, id, keysToo)` — вход по PIN-коду → `C_DestroyObject` по `CKA_ID` → перечитывание). По успеху снимок объектов обновляется (`updateNfcObjects`), поэтому список сразу отражает удаление без повторного поднесения; при ошибке — «Начать заново». Вариант «+ключи» показывается, если ключ точно есть или его наличие ещё неизвестно (подключались без PIN-кода — вход при удалении всё равно откроет ключ). Иллюстрация и звуки — те же (`NfcHoldAnimation`, `Feedback`).

**Подключение по NFC ведёт на свойства токена.** После успешного подключения мастер (`NfcConnectPage`) открывает **экран свойств токена** (`TokenDetailsPage`, как у USB), а не сразу список сертификатов. Объекты уже считаны при подключении (`nfcRead` = вход + чтение) и сохранены в снимке, поэтому из деталей их видно **без повторного поднесения** (кнопка «Объекты токена»). На шаге «взять токен» есть кнопка **«Продолжить без PIN-кода»** — тогда читаются только публичные сертификаты (без входа, `preview`).

**Единое написание «PIN-код».** По всему интерфейсу и в сообщениях о результате используется одно написание — **PIN-код** (ранее встречались «PIN», «Пин», «пин»); латинское `PIN` + `-код`, со склонением по падежам.

Раскладка ABI-структур PKCS#11 (`CK_INFO`, `CK_SLOT_INFO`, `CK_TOKEN_INFO`, префикс `CK_FUNCTION_LIST`) задана с естественным выравниванием (без Windows-упаковки `pack(1)`) и проверяется `static_assert` отдельно для armv7hl (ILP32) и aarch64 (LP64); значения сверены с реальной официальной библиотекой Рутокен.

Бинарная библиотека не копируется внутрь MIT-пакета приложения. В релизе рядом публикуется **неизменённый официальный RPM Рутокен**: это сохраняет исходный пакет и соответствует условиям распространения неизменённого ПО из разделов 4.3–4.4 [лицензионного соглашения Рутокен](https://www.rutoken.ru/download/license/License_Agreement_Rutoken.pdf). Официальная страница загрузки: [PKCS#11-библиотека Рутокен](https://www.rutoken.ru/support/download/pkcs/).

## Подтверждено на физическом устройстве

На 32-битном телефоне с Авророй 5.2 версия 0.0.2 показала зелёные статусы `nfcd`, NFC-адаптера, `libpcsclite` и `pcscd`. При подключении Рутокен ЭЦП по USB в списке появляется дополнительный PC/SC-ридер `Aktiv Rutoken ECP 00`. Работа непосредственно через PKCS#11 начнётся в v0.0.3.

## Установка на телефон (Аврора 5.x)

1. **Включите режим разработчика** (Настройки → Приложения/Система → Режим разработчика; потребуется учётная запись Аврора ID или локальное подтверждение — зависит от версии ОС). Без него пакет, подписанный тестовым ключом, не установится (`BadPackageSignature`) или не запустится.
2. Разрешите установку недоверенных приложений (Настройки → Безопасность → Недоверенное ПО).
3. Скачайте из [релиза ci-latest](https://github.com/code-agent-43824/aurora-rutoken/releases/tag/ci-latest) **два** RPM своей архитектуры: для 32-битного устройства — оба `*.armv7hl.rpm`, для 64-битного — оба `*.aarch64.rpm` (пакет чужой архитектуры отклоняется с ошибкой `BadPackageArchitecture`).
4. Сначала откройте в «Файлах» и установите `ru.rutoken.librtpkcs11ecp-2.19.0.0-1.<архитектура>.rpm`, затем `ru.codeagent43824.rutokentestapp-0.3.0-1.<архитектура>.rpm`.
5. Запустите приложение. Подключите Рутокен по USB или поднесите к NFC-антенне — карточка токена появится в списке **сама**. Тапните по карточке → детали токена; сертификаты видны сразу (кнопка «Объекты токена»), а вход по PIN (PIN пользователя по умолчанию — `12345678`) добавляет ключи. Меню-шторка сверху: «Диагностика» и «Обновить». Альтернатива установке из «Файлов» — `devel-su pkcon install-local <путь-к-rpm>`.
