### v0.2 — Вход по PIN и детали токена — В РАБОТЕ с 2026-07-20

Цель: по тапу на карточку токена — экран деталей с расширенной информацией и вводом PIN пользователя; проверка PIN через `C_Login`, индикатор оставшихся попыток.

Архитектурное решение (конфликт контекстов): `C_Initialize`/`C_Finalize` глобальны для процесса; фоновый опрос `TokenWatcher` и операция логина не должны пересекаться. Решение — **общий `QMutex`** вокруг каждого изолированного цикла `init…finalize` (и опрос, и логин берут его). ### v0.7 — Запросы на сертификат (CSR) — В РАБОТЕ с 2026-07-23

Цель: сформировать запрос PKCS#10 (CSR) для существующей ключевой пары на токене — заполнить DN, подписать закрытым ключом **на токене** (ГОСТ), экспортировать в PEM. Подробности кодирования — `docs/RESEARCH.md` §5г. Подпись CSR переиспользует уже заведённый примитив `C_Sign` (№43/44) с механизмом «подпись с хешем» — новых ABI-функций не нужно, только константы механизмов. **Порядок байтов ключа/подписи ГОСТ в DER — device-uncertain, сверяем openssl+ГОСТ на устройстве.**

**Этап 1 (0.7.0-1) — формирование и экспорт CSR:**
- [ ] ABI: константы механизмов подписи-с-хешем `CKM_GOSTR3410_WITH_GOSTR3411_12_256`=0xD4321008, `_512`=0xD4321009 (vendor), `CKM_SHA256_RSA_PKCS`=0x40 (RSA). Новых указателей функций не требуется (переиспользуем `C_Sign`/`C_SignInit`)

Механизм/OID алгоритма — по `CKA_KEY_TYPE` найденного по `CKA_ID` открытого ключа
