# Инструкции для coding-агентов

Этот файл — канонический источник правил проекта для любых coding-агентов. Агент-специфичные файлы (например, `CLAUDE.md`) могут только дополнять его и не должны дублировать или переопределять эти правила.

## Проект

Тестовое приложение для Рутокен ЭЦП 3.0 по USB и NFC на ОС Аврора 5.2 и выше.

- Стек: Qt/C++ + QML, qmake, RPM.
- Application ID: `ru.codeagent43824.rutokentestapp`.
- Доступ к токену: `librtpkcs11ecp.so` (PKCS#11) → `libpcsclite`/`pcscd` → CCID (USB) или NFC-стек.
- Целевые архитектуры: `armv7hl` и `aarch64`; телефон владельца — `armv7hl`.
- Подробные решения и источники: `docs/RESEARCH.md`.
- Дорожная карта и текущий статус: `PLAN.md`.
- Хронология выполненных действий: `docs/JOURNAL.md`.

## Обязательный порядок работы

1. Перед любым действием прочитать `PLAN.md` и верхнюю актуальную запись `docs/JOURNAL.md`.
2. Выполнить `git fetch origin main` и убедиться, что локальная `main` не отстаёт; учитывать изменения других агентов.
3. До изменения кода, CI, конфигурации или структуры записать конкретное действие в `PLAN.md` со статусом `[ ]`/`[~]` и при необходимости пояснить его в `docs/JOURNAL.md`.
4. Только после документирования выполнять действие.
5. Провести проверки, соответствующие риску изменения. Для сборочных изменений дождаться GitHub Actions и проверить фактические логи/артефакты, а не только зелёный статус.
6. После действия обновить `PLAN.md` фактическим статусом `[x]`/`[!]`, добавить верхнюю запись в `docs/JOURNAL.md` и обновить `docs/RESEARCH.md`, если появились новые факты об Авроре или Рутокен.
7. Не оставлять README, план, журнал, research, версию spec и фактическое поведение рассинхронизированными.

## Git

- Единственная рабочая ветка — `main`.
- Не создавать ветки и pull requests.
- Каждый завершённый логический шаг коммитить напрямую в `main` и сразу отправлять `git push origin main`.
- Перед коммитом проверять `git diff --check` и `git status`; не включать чужие или сгенерированные файлы.
- Использовать обычные прямые команды git. GitHub API разрешён только для чтения статусов, логов, релизов и артефактов.

## Сборка и выпуск RPM

- RPM собираются в `.github/workflows/build-rpm.yml` с Aurora Platform SDK `5.2.1.200`.
- armv7hl и aarch64 обязаны собираться в независимых чистых matrix jobs. Один job — одна архитектура (`TARGET_ARCH`).
- `ci/build-rpm.sh` обязан подписать пакет официальным публичным тестовым ключом OMP и запустить `ci/verify-rpm.sh`.
- Проверки каждого пакета обязательны: RPM arch, ELF machine, dynamic loader и наличие ожидаемой подписи OMP Regular.
- `ci-latest` публикуется только после успеха и проверки обеих архитектур.
- Пакеты с публичным тестовым ключом предназначены только для режима разработчика.
- **Доставка: релиз `ci-latest` с голыми `.rpm` — ОСНОВНОЙ канал, обязателен.** CI обязан лить `.rpm` прямо в релиз из сборочных jobs (отдельные файлы рядом, без архива). Этот канал **убирать нельзя**. Артефакты `actions/upload-artifact` (per-arch) — **второй** канал по просьбе владельца. Артефакты Actions по природе **всегда zip** (ограничение GitHub, обойти нельзя); владелец это принял (решение 2026-07-20) и выбрал оставить их. **Вопрос закрыт: не пытаться «раззиповать» артефакты и не убирать их; релиз с голыми RPM не трогать.**
